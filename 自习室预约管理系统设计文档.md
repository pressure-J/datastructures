## 一、问题描述

本实验设计并实现一个面向高校图书馆的自习室预约管理系统。系统要支持学生每日预约座位、凭预约记录扫码签到签退、严格执行“1人1桌1座”、空闲超时自动释放座位，并能够统计座位使用情况。

系统设计目标包括：

1. 使用 C 语言完成主要数据结构和业务模块设计。
2. 使用链表、哈希表、队列、栈、最小堆等结构支撑查询、预约、等待和超时处理。
3. 通过统一错误码和函数封装降低模块耦合。
4. 支持学生、座位、预约等数据的持久化扩展。
5. 形成可编译、可测试、可继续扩展的课程实践项目。

## 二、系统分析

### 1. 用户需求（从使用者的视角）

学生用户需要完成注册登录、查询可用座位、创建预约、取消预约、签到、签退、查询个人记录和信用分等操作。学生侧的关键要求是查询结果准确、冲突提示明确、预约状态可追踪。

管理用户需要维护自习室、座位、开放时间、超时阈值和统计信息。管理侧的关键要求是能及时释放长期空闲座位，减少资源浪费，并通过统计数据了解座位使用情况。

系统运行环境为普通 Windows 或 Linux 命令行环境，使用 C99/C11 标准编译。数据量按课程实验规模设计，但结构上保留扩展空间。

### 2. 数据结构设计与实现

系统数据结构选型如下。

![数据结构选型与作用示意图](assets/data_structure_selection.png)

#### 2.1 链表

链表用于保存动态数量的数据节点和查询结果。学生、座位、预约结构体中都包含 `next` 指针，可以通过头插或尾插形成链表。链表的优点是插入删除方便，适合课程实验中动态数据规模不固定的场景；缺点是按编号查找需要遍历，因此需要配合哈希表使用。

#### 2.2 哈希表

学生、座位、预约分别设计独立哈希表，键分别为学号、座位编号和预约编号。哈希函数使用 BKDR 思路，将字符串映射到固定大小的桶数组中；冲突采用链地址法，每个桶保存一条冲突链。该结构能将高频查询平均时间复杂度降低到 O(1)。

#### 2.3 队列、栈和最小堆

等待队列用于处理某个座位暂不可用时的等待请求，遵循 FIFO 原则。操作栈用于保存状态变化前后的操作记录，便于撤销或调试。最小堆用于超时事件调度，每次优先检查最早到期的预约，避免频繁全表扫描。

#### 2.4 文件存储

当前项目已包含 `students.csv` 和 `seats.csv` 测试数据。完整系统可继续增加 `reservations.csv`、`rooms.csv` 和 `config.ini`。读文件时应逐行解析并校验字段长度；写文件时应先写入临时文件，确认成功后再替换原文件，以减少异常中断造成的数据损坏。

### 3. 算法设计

#### 3.1 哈希查找算法

哈希查找先对键字符串计算桶下标，再遍历对应桶链表。若找到键相同的节点，则返回数据指针或复制结果；若遍历结束仍未找到，则返回对应错误码。插入时先检查是否已存在相同键，避免重复学生、重复座位或重复预约编号。

#### 3.2 预约冲突检测算法

预约冲突检测包括两类：同一座位同一时间段不能被重复预约，同一学生同一时间段不能预约多个座位。判断区间相交的条件是：新开始时间早于已有结束时间，且新结束时间晚于已有开始时间。只要已有预约状态仍为待签到或已签到，就认为该时间段被占用。

#### 3.3 预约、签到与超时流程

![预约、签到与超时处理流程图](assets/reservation_checkin_timeout_flow.png)

创建预约时，系统按“验证学生 -> 验证座位 -> 验证时间 -> 检测冲突 -> 写入预约 -> 更新座位状态”的顺序执行。签到时只允许待签到状态进入已签到状态，并记录签到时间与最后活跃时间。签退时只允许已签到状态进入已完成状态，并释放座位。超时检测发现空闲超过 30 分钟后，将预约置为超时失效，释放座位并扣除信用分。

#### 3.4 链表排序算法

预约记录按开始时间输出时，可以对预约链表进行归并排序。归并排序适合链表结构，不需要随机访问数组下标，时间复杂度为 O(n log n)。排序结果可用于个人预约历史、座位预约历史和统计分析。

## 五、系统总结与比较分析

### 1. 系统总结

本系统方案围绕自习室预约管理的完整业务流程展开，将学生、座位、预约、自习室和超时事件抽象为数据对象，并用数据结构课程中的典型结构支撑业务功能。当前已完成并可运行的部分包括链表、哈希表、学生管理和座位管理，为后续预约、超时和统计模块提供了基础接口。

从正确性角度看，统一枚举状态和错误码有利于避免魔法数字和模糊返回值；从效率角度看，哈希表提升了编号查询效率，最小堆方案能降低超时处理成本；从可维护性角度看，按头文件分模块封装便于团队成员并行开发。

### 2. 比较研究

若使用数组顺序表保存所有学生、座位和预约记录，结构简单、实现成本低，但插入删除和按编号查询效率较差，数据规模增大后需要频繁移动元素或线性扫描。若使用平衡树或跳表，可以支持有序查询和范围查询，但实现复杂度超出本实验主要要求。

本系统采用“哈希表 + 链表”的组合，适合课程实践中按编号频繁查询、动态增加记录的场景；对于超时事件，采用最小堆比每次全表扫描更适合大量预约记录。综合比较后，该方案在实现难度和运行效率之间较为平衡。
